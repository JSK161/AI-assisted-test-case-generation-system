# AI辅助测试用例生成系统需求规格说明书

## 1. 文档概述

### 1.1 项目名称

AI辅助测试用例生成系统。

### 1.2 项目背景

在软件测试过程中，测试人员通常需要根据需求文档、接口说明和业务流程人工编写测试用例。该过程存在编写效率低、用例格式不统一、异常场景遗漏、覆盖情况难以统计等问题。本项目面向专业实训场景，设计并实现一个基于 DeepSeek API 的 AI辅助测试用例生成系统，帮助用户根据需求描述自动生成结构化测试用例，并支持人工采纳、修改和统计分析。

### 1.3 建设目标

基础层建设目标如下：

1. 实现用户登录或身份识别。
2. 实现测试项目管理。
3. 支持输入软件需求描述。
4. 调用 DeepSeek API 自动生成结构化测试用例。
5. 将项目、需求、测试用例和采纳状态持久化到 MySQL。
6. 支持测试用例查看、编辑和采纳状态维护。
7. 提供测试用例数量、采纳数量和采纳率统计。
8. 提供基本异常提示和部署说明。

## 2. 项目范围

### 2.1 基础层范围

基础层是本项目必须完成的核心交付范围，主要用于满足实训合格要求。系统需要形成完整的业务闭环：

用户登录 -> 创建测试项目 -> 输入需求描述 -> AI生成测试用例 -> 保存测试用例 -> 查看和维护测试用例 -> 统计采纳率。

### 2.2 暂不纳入基础层范围

以下功能作为提高层或挑战层扩展，不作为基础层强制要求：

1. 知识库/RAG。
2. 多模型切换。
3. 流式输出。
4. 多模型回答对比。
5. 自动化测试脚本生成。
6. 复杂权限管理。
7. 性能优化和复杂安全控制。

## 3. 用户角色

### 3.1 组长/管理员

组长或管理员可以登录系统，创建项目，查看项目数据，生成测试用例，维护采纳状态，并查看统计结果。

### 3.2 小组成员

小组成员可以登录系统，进入项目，录入需求，生成测试用例，查看和修改测试用例。

基础层可以先使用简单角色字段区分管理员和普通成员，不做复杂权限控制。

## 4. 功能性需求

### 4.1 用户登录

系统应提供账号密码登录功能。

功能要求：

1. 用户输入账号和密码。
2. 系统校验账号密码是否正确。
3. 登录成功后进入项目列表页。
4. 登录失败时提示账号或密码错误。
5. 登录状态应能在前端保存，用于访问后续页面。

验收标准：

1. 正确账号密码可以登录。
2. 错误账号密码不能登录。
3. 未登录用户不能直接进入主要业务页面。

### 4.2 项目管理

系统应支持测试项目的创建和查询。

功能要求：

1. 用户可以创建项目。
2. 项目信息包括项目名称、项目描述、创建人、创建时间。
3. 用户可以查看项目列表。
4. 用户可以进入项目详情页。

验收标准：

1. 创建项目后，项目数据保存到 MySQL。
2. 项目列表可以展示已创建项目。
3. 点击项目可以进入对应项目详情。

### 4.3 需求录入

系统应支持录入待测试的软件需求文本。

功能要求：

1. 用户选择项目后录入需求标题和需求描述。
2. 需求描述不能为空。
3. 需求信息保存到 MySQL。
4. 系统可以展示某个项目下的需求列表。

验收标准：

1. 空需求无法提交，并显示提示信息。
2. 有效需求提交后可以在页面中查看。
3. 需求记录与项目正确关联。

### 4.4 AI测试用例生成

系统应基于需求描述调用 DeepSeek API 生成测试用例。

功能要求：

1. 用户选择某条需求后点击生成测试用例。
2. 后端根据需求内容构造 Prompt。
3. 后端调用 DeepSeek API。
4. AI返回结构化 JSON 格式测试用例。
5. 后端解析 JSON 并保存测试用例。
6. 前端展示生成结果。

测试用例字段应包括：

1. 用例标题。
2. 用例类型。
3. 前置条件。
4. 测试步骤。
5. 测试数据。
6. 预期结果。
7. 优先级。
8. 采纳状态。

验收标准：

1. 输入有效需求后可以生成测试用例。
2. 生成结果具有固定结构。
3. 生成结果可以保存到 MySQL。
4. DeepSeek API 调用失败时系统给出错误提示。

### 4.5 测试用例管理

系统应支持查看、编辑和维护测试用例。

功能要求：

1. 用户可以查看项目下所有测试用例。
2. 用户可以查看测试用例详情。
3. 用户可以修改测试用例标题、步骤、测试数据、预期结果、优先级等信息。
4. 用户可以设置采纳状态。
5. 采纳状态包括：未处理、已采纳、需修改、已拒绝。

验收标准：

1. 测试用例列表展示正确。
2. 修改测试用例后数据库同步更新。
3. 修改采纳状态后统计数据同步变化。

### 4.6 采纳率统计

系统应提供基础统计能力。

功能要求：

1. 统计项目下测试用例总数。
2. 统计已采纳测试用例数量。
3. 统计需修改测试用例数量。
4. 统计已拒绝测试用例数量。
5. 计算采纳率。

采纳率计算公式：

```text
采纳率 = 已采纳测试用例数量 / 测试用例总数 * 100%
```

验收标准：

1. 统计数据与数据库记录一致。
2. 修改采纳状态后统计结果更新。
3. 当测试用例总数为 0 时，采纳率显示为 0%。

## 5. 非功能性需求

### 5.1 易用性

系统页面应简洁清晰，核心流程不超过以下步骤：

登录 -> 创建项目 -> 输入需求 -> 生成测试用例 -> 查看结果。

### 5.2 可靠性

系统应对常见异常进行提示，包括：

1. 登录失败。
2. 必填项为空。
3. 数据库保存失败。
4. DeepSeek API 调用失败。
5. AI返回内容格式不正确。

### 5.3 可维护性

后端应采用分层结构，包括 Controller、Service、Mapper、Entity、DTO。前端应按页面和接口模块拆分，避免所有逻辑集中在单个文件中。

### 5.4 安全性

DeepSeek API Key 不应写入前端代码，也不应提交到 GitHub。API Key 应通过后端配置文件或环境变量读取。

## 6. 技术栈要求

1. 后端：Java + Spring Boot。
2. 前端：Vue 3 + Vite。
3. 数据库：MySQL。
4. AI能力：DeepSeek API。
5. ORM：MyBatis-Plus。
6. UI组件库：Element Plus。
7. HTTP请求：Axios。
8. 图表展示：ECharts。

## 7. 接口需求概览

基础层接口如下：

```text
POST /api/auth/login
POST /api/projects
GET  /api/projects
GET  /api/projects/{id}

POST /api/requirements
GET  /api/requirements/project/{projectId}

POST /api/ai/generate-test-cases
GET  /api/test-cases/project/{projectId}
GET  /api/test-cases/{id}
PUT  /api/test-cases/{id}
PUT  /api/test-cases/{id}/adoption-status

GET  /api/statistics/project/{projectId}
```

## 8. 数据需求概览

基础层需要以下数据表：

1. 用户表 `sys_user`：保存登录账号、密码、角色。
2. 项目表：保存项目基本信息。
3. 需求表：保存需求标题和需求描述。
4. 测试用例表：保存 AI 生成和人工修改后的测试用例。
5. AI生成记录表：保存生成时间、输入内容、模型名称和调用状态。

## 9. 验收标准

基础层最终验收应满足：

1. 系统可以正常启动。
2. 用户可以登录。
3. 用户可以创建项目。
4. 用户可以录入需求。
5. 系统可以调用 DeepSeek API 生成测试用例。
6. 测试用例可以保存到 MySQL。
7. 用户可以查看、修改和标记测试用例采纳状态。
8. 系统可以展示采纳率统计。
9. README 中包含部署说明、数据库初始化说明和 DeepSeek API Key 配置说明。
