# AI辅助测试用例生成系统需求规格说明书

## 1. 文档概述

### 1.1 项目名称

AI辅助测试用例生成系统。

### 1.2 项目背景

在软件测试工作中，测试人员通常需要根据需求说明、接口文档和业务流程人工编写测试用例。人工编写方式存在效率较低、测试场景容易遗漏、用例结构不统一、经验依赖较强等问题。随着大语言模型能力的发展，可以利用 AI 对需求内容进行理解、追问和结构化输出，从而辅助测试人员更快地形成较完整的测试用例方案。

本项目面向软件工程专业实训场景，设计并实现一个基于 DeepSeek API 的 AI辅助测试用例生成系统。用户输入想要测试的模块或功能后，系统先调用 DeepSeek 生成针对该模块的动态反问问题，用户选择回答后，系统再调用 DeepSeek 生成包含测试范围、风险提示和测试用例列表的结构化方案。

### 1.3 建设目标

本系统的建设目标如下：

1. 提供简洁的 AI 对话式页面，支持用户输入待测试模块或功能。
2. 调用 DeepSeek API，根据用户输入动态生成补充问题。
3. 支持用户选择问题答案或补充自定义内容。
4. 根据原始需求、AI 反问问题和用户回答生成结构化测试用例方案。
5. 展示被测对象与范围、风险提示、测试用例列表等内容。
6. 支持复制方案和导出 Markdown 文件。
7. 后端统一管理 DeepSeek API Key，避免密钥暴露到前端或 GitHub。
8. 保留基础用户、项目、需求、测试用例和统计接口，为后续扩展提供支撑。

## 2. 项目范围

### 2.1 当前实现范围

当前项目以 AI 对话式测试用例生成为主线，核心业务流程为：

```text
用户输入测试模块或功能
  -> 后端调用 DeepSeek 生成动态反问问题
  -> 前端展示问题和选项
  -> 用户选择或补充答案
  -> 后端调用 DeepSeek 生成测试用例方案
  -> 前端展示结果并支持复制、导出
```

### 2.2 基础支撑范围

系统同时保留以下基础支撑功能，作为后续扩展模块：

1. 用户登录接口。
2. 项目管理接口。
3. 需求管理接口。
4. 测试用例管理接口。
5. 采纳状态维护接口。
6. 项目统计接口。
7. MySQL 数据库初始化脚本。

### 2.3 暂不纳入当前版本范围

以下功能可作为提高层或挑战层扩展：

1. 知识库/RAG 检索增强。
2. 多模型切换和多模型回答对比。
3. 流式输出。
4. 复杂权限管理。
5. 自动化测试脚本直接生成和执行。
6. 在线协同编辑测试用例。
7. 复杂安全审计和性能优化。

## 3. 用户角色

### 3.1 组长/管理员

组长负责系统整体运行、GitHub 仓库管理、DeepSeek API 配置、功能联调和最终演示。组长可以使用系统完成 AI 测试用例生成流程，并负责组织小组成员完成各模块文档。

### 3.2 小组成员

小组成员可以围绕各自负责模块使用系统，包括登录模块、需求输入与动态问答模块、测试用例结果展示模块、测试与质量保障模块、文档与部署模块等。

### 3.3 普通使用者

普通使用者是系统主要面向对象，可以在页面输入测试需求，通过 AI 动态问答补充信息，并获取测试用例方案。

## 4. 功能性需求

### 4.1 AI 对话式首页

系统应提供 AI 对话式首页，用于输入测试需求。

功能要求：

1. 页面展示系统名称和输入区域。
2. 用户可以输入要测试的模块、功能或接口。
3. 用户可以选择推荐分类，如 Web 测试、接口测试、安全测试、自动化脚本等。
4. 用户可以填写可选参考 URL。
5. 点击发送后进入动态问答流程。

验收标准：

1. 页面能够正常打开。
2. 输入为空时不能进入下一步，并给出提示。
3. 输入有效需求后可以进入 AI 动态反问流程。

### 4.2 DeepSeek 动态反问

系统应根据用户输入调用 DeepSeek 生成动态反问问题，而不是使用固定问题模板。

功能要求：

1. 前端将用户输入的需求描述提交给后端。
2. 后端通过 `/api/chat/questions` 调用 DeepSeek。
3. DeepSeek 根据不同模块生成不同问题，例如登录模块、退款模块、支付模块应生成不同反问。
4. 每个问题包含问题标题、问题类型、候选选项和选项说明。
5. 问题类型支持单选和多选。
6. 系统自动追加“总览并确认”步骤。

动态反问返回字段应包括：

1. 问题 ID。
2. 问题标题。
3. 问题类型。
4. 是否允许自定义补充。
5. 选项列表。
6. 使用模型名称。

验收标准：

1. 输入“登录模块”时，系统生成与账号、密码、验证码、登录安全相关的问题。
2. 输入“订单退款模块”时，系统生成与退款审核、原路退回、退款失败、订单状态回滚相关的问题。
3. 前端可以正确展示 DeepSeek 返回的问题。
4. DeepSeek 调用失败时，系统可以使用本地预览问题继续演示。

### 4.3 用户回答收集

系统应支持用户回答 AI 生成的动态问题。

功能要求：

1. 用户可以选择单选或多选答案。
2. 用户可以在允许自定义的问题中补充其他内容。
3. 系统记录每个问题的用户选择。
4. 系统在左侧区域汇总已选择的核心场景。
5. 用户可跳过部分问题。
6. 用户确认后进入测试用例生成阶段。

验收标准：

1. 用户选择答案后可以进入下一题。
2. 多选问题可以选择多个选项。
3. 自定义补充内容可以进入最终 Prompt。
4. 总览确认后可以生成测试用例方案。

### 4.4 DeepSeek 测试用例生成

系统应根据用户输入和回答调用 DeepSeek 生成测试用例方案。

功能要求：

1. 前端调用 `/api/chat/generate` 提交原始需求、用户回答和参考 URL。
2. 后端构造测试用例生成 Prompt。
3. 后端调用 DeepSeek API。
4. DeepSeek 返回严格 JSON 格式内容。
5. 后端解析生成结果并返回前端。
6. DeepSeek 调用失败时，系统提供本地预览结果。

生成结果字段应包括：

1. 方案标题。
2. 方案摘要。
3. 被测对象与范围。
4. 风险提示。
5. 测试用例列表。
6. 模型名称。

单条测试用例字段应包括：

1. 用例编号。
2. 用例标题。
3. 优先级。
4. 用例类型。
5. 前置条件。
6. 测试步骤。
7. 预期结果。

验收标准：

1. 输入有效需求并完成问答后可以生成测试方案。
2. 生成结果不少于若干条测试用例，覆盖正常、异常、边界、安全、兼容性等场景。
3. 页面显示使用模型，如 `deepseek-chat`。
4. 生成内容具有较清晰的测试步骤和预期结果。

### 4.5 结果展示

系统应将生成结果以文档化形式展示。

功能要求：

1. 展示测试方案标题。
2. 展示被测对象与范围表格。
3. 展示测试用例列表。
4. 展示风险提示。
5. 显示测试用例编号、优先级、类型、前置条件、步骤和预期结果。
6. 支持页面滚动查看完整内容。

验收标准：

1. 结果页面内容完整。
2. 表格和用例列表排版清晰。
3. 桌面端和移动端不出现明显布局错乱。

### 4.6 方案复制与导出

系统应支持将生成方案复制或导出。

功能要求：

1. 用户可以点击“复制方案”按钮。
2. 系统将当前方案转换为 Markdown 文本。
3. 用户可以点击“导出 Markdown”按钮下载 `.md` 文件。
4. 导出文件名与当前测试主题相关。

验收标准：

1. 点击复制后剪贴板中包含方案内容。
2. 导出的 Markdown 文件包含标题、范围、测试用例和风险提示。

### 4.7 参考 URL 内容读取

系统应支持可选参考 URL。

功能要求：

1. 用户可输入公开可访问的需求页面或案例页面 URL。
2. 后端读取 URL 页面内容。
3. 后端清洗 HTML 标签，截取有效文本。
4. 参考内容参与 DeepSeek 动态反问和测试用例生成。
5. URL 读取失败不阻断主流程。

验收标准：

1. 有效 URL 可以被后端读取并加入 Prompt。
2. 无效 URL 不导致系统崩溃。

### 4.8 基础登录与项目接口

系统保留基础登录、项目、需求、测试用例和统计接口。

功能要求：

1. 支持默认用户登录。
2. 支持创建和查询项目。
3. 支持创建和查询需求。
4. 支持生成并管理持久化测试用例。
5. 支持采纳状态维护和统计。

验收标准：

1. 相关接口可正常响应。
2. 数据库初始化脚本可创建基础表。
3. 后续可在当前 AI 对话主流程基础上扩展项目化管理。

## 5. 非功能性需求

### 5.1 易用性

系统首页应直接呈现可用的 AI 输入体验，不使用复杂菜单作为入口。核心流程应保持简洁：

```text
输入需求 -> AI 反问 -> 选择回答 -> 生成方案 -> 复制或导出
```

### 5.2 可靠性

系统应对以下异常进行处理：

1. 输入为空。
2. DeepSeek API Key 未配置。
3. DeepSeek API 调用失败。
4. DeepSeek 返回内容不是合法 JSON。
5. URL 读取失败。
6. 后端接口异常。

### 5.3 安全性

DeepSeek API Key 只能配置在后端环境变量中，不应写入前端代码、后端源码、README 示例真实值或 GitHub 仓库。

### 5.4 可维护性

后端采用 Controller、Service、DTO、VO、Config、Exception 等分层结构。前端采用 API、View、Utils、Types 等目录拆分，便于后续扩展。

### 5.5 可测试性

系统应提供前端工具函数测试和后端 Prompt/Parser 测试，保证动态反问解析、测试方案解析和前端问答流程可验证。

## 6. 技术栈要求

1. 后端：Java 17、Spring Boot 3。
2. 前端：Vue 3、Vite、TypeScript。
3. 数据库：MySQL 8。
4. AI 接口：DeepSeek API。
5. ORM：MyBatis-Plus。
6. 前端组件与图标：Element Plus、lucide 图标。
7. HTTP 请求：Axios。
8. 构建与测试：Maven、Vitest、vue-tsc。

## 7. 接口需求概览

AI 对话主流程接口：

```text
POST /api/chat/questions
POST /api/chat/generate
```

基础支撑接口：

```text
POST /api/auth/login
POST /api/projects
GET  /api/projects
GET  /api/projects/{id}

POST /api/requirements
GET  /api/requirements/project/{projectId}

POST /api/ai/generate-test-cases
GET  /api/test-cases/project/{projectId}
GET  /api/test-cases/{id}
PUT  /api/test-cases/{id}
PUT  /api/test-cases/{id}/adoption-status

GET  /api/statistics/project/{projectId}
```

## 8. 数据需求概览

基础数据库表包括：

1. 用户表 `sys_user`：保存用户账号、密码、角色等信息。
2. 项目表 `project`：保存测试项目基本信息。
3. 需求表 `requirement`：保存需求标题和需求描述。
4. 测试用例表 `test_case`：保存测试用例内容和采纳状态。
5. AI 生成记录表 `ai_generation_record`：保存 AI 调用记录。

当前 AI 对话式生成结果主要在前端展示和导出，后续可扩展为与项目、需求、测试用例表关联保存。

## 9. 验收标准

最终验收应满足：

1. 系统可以正常启动。
2. 前端页面可以输入测试模块或功能。
3. 系统可以调用 DeepSeek 生成动态反问问题。
4. 用户可以选择回答并确认生成。
5. 系统可以调用 DeepSeek 生成结构化测试用例方案。
6. 页面可以展示测试范围、风险提示和测试用例列表。
7. 支持复制方案和导出 Markdown。
8. DeepSeek API Key 不出现在 GitHub 仓库中。
9. README 中包含启动、数据库初始化和 DeepSeek 配置说明。
10. 前端测试、后端测试和前端构建能够通过。
